*Version: 1.0 — 2026-03-09*
*Status: Draft*
*Authors: Serj & Scaff*
*Ref: `hipocampus-architecture.md` (Layer 2: Foreground), `cortex-architecture.md` (§4: Context Window)*

---

## 1. Problem Statement

Cortex maintains a single unified session (`cortex_session`) across all channels. As conversations accumulate, the foreground — verbatim message history — grows unbounded. Without intervention, it fills the entire context window, eventually exceeding the model's token limit and killing Cortex.

A naive soft cap (e.g., "keep last 20 messages") solves the token problem but creates a worse one: it cuts conversations mid-topic. The LLM receives the tail end of a discussion without the setup, leading to confused responses and lost context.

**The requirement:** Foreground must be bounded by a token budget, but the cut must never happen inside a coherent conversation block.

---

## 2. Core Concept: Shards

A **shard** is a group of consecutive messages in `cortex_session` that belong to a single coherent topic. A shard is the atomic unit of foreground context — it is either fully included or fully excluded. Never partially.

Examples of shards:
- "Cortex death loop debugging" — 45 messages across 30 minutes
- "Token monitor TASK column fix" — 80 messages across 2 hours
- "Quick weather question" — 3 messages, 30 seconds
- "Code search tool implementation" — 30 messages across 1 hour

A shard has:
- A **boundary** — the point where one topic ends and another begins
- A **token cost** — the total tokens of all messages in the shard
- A **time range** — first and last message timestamps
- A **topic label** — short human-readable description (generated by the boundary detector)

## 3. Data Model

### 3.1 Shard Metadata Table

```sql
CREATE TABLE IF NOT EXISTS cortex_shards (
  id TEXT PRIMARY KEY,                    -- UUID
  channel TEXT NOT NULL,                  -- source channel (whatsapp, webchat, etc.)
  topic TEXT NOT NULL,                    -- short label: "Token monitor fix"
  first_message_id INTEGER NOT NULL,      -- FK to cortex_session.id
  last_message_id INTEGER NOT NULL,       -- FK to cortex_session.id (updated as shard grows)
  token_count INTEGER NOT NULL DEFAULT 0, -- approximate tokens in this shard
  message_count INTEGER NOT NULL DEFAULT 0,
  started_at TEXT NOT NULL,               -- timestamp of first message
  ended_at TEXT,                          -- timestamp of last message (NULL = open/active shard)
  created_at TEXT NOT NULL,               -- when the shard record was created
  created_by TEXT NOT NULL DEFAULT 'gardener' -- 'gardener' | 'inline' | 'manual'
);

CREATE INDEX IF NOT EXISTS idx_shards_channel ON cortex_shards(channel);
CREATE INDEX IF NOT EXISTS idx_shards_ended ON cortex_shards(ended_at);
```

### 3.2 Message-to-Shard Mapping

Each message in `cortex_session` gets a `shard_id` column:

```sql
ALTER TABLE cortex_session ADD COLUMN shard_id TEXT REFERENCES cortex_shards(id);
```

Messages without a `shard_id` (NULL) are **unsharded** — either too new for the Gardener to process, or from before sharding was enabled. Context assembly treats consecutive unsharded messages at the tail as the **active implicit shard**.

### 3.3 Raw Data Unchanged

`cortex_session` retains all messages indefinitely. Shards are metadata layered on top — a slicing lens, not a transformation. Re-sharding with improved logic is always possible without data loss.

## 4. Shard-Based Foreground Assembly

### 4.1 Token Budget

The foreground has a configurable token cap with a ±20% tolerance band:

```json
{
  "hippocampus": {
    "foreground": {
      "tokenCap": 8000,
      "tolerancePct": 20
    }
  }
}
```

Effective range for an 8,000-token cap at 20% tolerance: **6,400 – 9,600 tokens**.

### 4.2 Assembly Algorithm

Context assembly builds the foreground bottom-up, from newest to oldest:

```
1. Start with running_total = 0
2. Load the active shard (the currently open shard, ended_at = NULL)
   - Add its token count to running_total
   - Always included (this is what the user is currently discussing)
3. Walk backward through closed shards (newest-first):
   a. If running_total + shard.token_count <= cap * (1 + tolerance):
      - Include the shard, add to running_total
   b. Else:
      - Stop. This shard and all older ones are excluded.
4. Return included shards as the foreground context.
```

**The active shard is always included**, even if it alone exceeds the cap. This is a deliberate design choice — you cannot cut the current conversation. The cap governs how much *history* is retained alongside it. In practice, the token threshold (Tier 1B) prevents the active shard from growing unbounded — it forces a split before it gets too large.

### 4.3 Example

```
Foreground cap: 8,000 tokens | Tolerance: 20% | Ceiling: 9,600

Shard 6 (active):      2,100 tokens → included  [running: 2,100]
Shard 5 (closed):      3,200 tokens → included  [running: 5,300]
Shard 4 (closed):      2,800 tokens → included  [running: 8,100]
Shard 3 (closed):      4,500 tokens → skip      [8,100 + 4,500 = 12,600 > 9,600]

Foreground: Shard 6 + 5 + 4 = 8,100 tokens
Shards 3, 2, 1: excluded from context, key facts in Hot Memory, verbatim retrievable via fetch_chat_history
```

### 4.4 Shard Separator in Context

Each shard boundary is marked in the assembled context with a lightweight separator:

```
--- [Topic: Code search tool implementation | ~1h ago | 30 messages] ---
<verbatim messages from shard>
--- [Topic: Executor sandbox fix | ~15min ago | 12 messages] ---
<verbatim messages from shard>
<unsharded active messages>
```

This gives the LLM awareness of topic boundaries and rough time distances without consuming significant tokens.

### 4.5 Cross-Channel Shards

Shards are per-channel. When assembling foreground for a WhatsApp-triggered turn, only WhatsApp shards are included. Other channels appear as Background summaries (Layer 3). If Cortex needs webchat history, it uses `fetch_chat_history` with a shard reference.

## 5. Shard Boundary Detection

Topic shifts in natural conversation are fuzzy — conversations meander, topics interleave, quick tangents interrupt longer discussions. Rather than relying on a single heavy mechanism, shard boundaries are detected via a **cascading strategy** that runs inline (near real-time), not deferred to the Gardener.

### 5.1 Principle: Inline Assignment

Every message is assigned to a `shard_id` on arrival. It joins the currently open (active) shard. Boundary detection runs asynchronously right behind the active conversation — by the time the Gardener sweeps, shards are already defined. The Gardener's role is to consume closed shards for fact extraction into Hot/Long Memory, not to create shard boundaries.

### 5.2 Cascading Detection Strategy

Boundary detection uses a cascade: cheap heuristics first, semantic checks only when heuristics don't trigger.

#### Tier 1: Heuristic Triggers (Fast & Free)

These run on every message with zero LLM cost:

**A. Time Gap:**
A pause of ≥ `timeGapMinutes` (default: 20-30 minutes) between consecutive messages is an automatic shard boundary. People naturally reset context after a break. The current shard is closed, and the new message starts a fresh shard.

**B. Token Threshold (Active Shard Bloat):**
If the currently open shard exceeds `maxShardTokens` (default: 8,000 tokens), it triggers a forced split. This prevents any single shard from growing unbounded during a long continuous conversation. The split point is the current message — the oversized shard is closed, and a new shard begins.

These two heuristics catch ~80% of natural boundaries for free.

#### Tier 2: Semantic Triggers (Smart & Cheap)

When a conversation flows continuously without time gaps or token overflows but shifts topics (e.g., moving from the token monitor to the executor sandbox), heuristics miss it.

**Sliding Window Check:**
Every `semanticCheckInterval` messages (default: 5-10), run a lightweight model (Haiku/Gemini Flash) with a focused binary prompt:

```
Given these recent messages, did the primary task or topic shift significantly?
If yes: return the message ID where the new topic begins and a 3-5 word label for each topic.
If no: return "no_shift".
Do NOT split on minor tangents that return to the main topic within 2-3 messages.
```

If a shift is detected, close the current shard (with the label for the old topic) and open a new one (with the label for the new topic). This generates the **topic label** at detection time — no separate labeling pass needed.

**Cost:** One Haiku call every 5-10 messages. Negligible compared to the Opus/Sonnet calls Cortex makes every turn.

#### Tier 3: Explicit Markers (Supplementary)

The user or Cortex can explicitly mark a shard boundary. Cortex can decide "this topic is done, starting a new one" and insert a boundary. The user could say "new topic" or similar. Available but not required — the cascade handles it automatically.

### 5.3 The Mega-Shard Problem

Even with the token threshold forced split, a long continuous session on one topic can produce a large shard. The token threshold (Tier 1B) already prevents unbounded growth by forcing a split at `maxShardTokens`. The result is sequential sub-shards that share the same topic but are sized for the budget:

```
Shard: "Token monitor implementation (1/3)" — messages 1-30, ~7,800 tokens
Shard: "Token monitor implementation (2/3)" — messages 31-55, ~7,200 tokens
Shard: "Token monitor implementation (3/3)" — messages 56-80, ~6,500 tokens
```

Context assembly treats sub-shards individually. Including (3/3) and (2/3) but excluding (1/3) is acceptable because the split happened at the token boundary, and Hot Memory holds the key facts from (1/3).

### 5.4 Topic Labels

Topic labels are generated at boundary detection time:
- **Heuristic boundaries (time gap):** Label is generated by a single cheap Haiku call on the closed shard's messages after the boundary fires. Async — doesn't block the loop.
- **Heuristic boundaries (token overflow):** Same — async Haiku call for labeling.
- **Semantic boundaries:** Label comes from the same detection call that identified the shift. No extra cost.
- **Explicit boundaries:** User or Cortex provides the label, or a Haiku call generates one.

## 6. Integration with Existing Systems

### 6.1 fetch_chat_history (Layer 2 Expansion)

Currently retrieves messages by channel + limit. With sharding, it gains an additional mode:

```
{ channel: "whatsapp", shard_id: "abc-123" }
```

Returns all messages in the specified shard. This is how Cortex pulls back excluded shards when it needs older context. The LLM sees it as a coherent block, not random rows.

The existing `{ channel, limit, before }` mode continues to work for raw chronological access.

### 6.2 Gardener Role (Changed)

With inline shard assignment, the Gardener's role changes. It no longer creates shard boundaries — those are handled by the cascading detection system (§5). The Gardener **consumes** closed shards:

1. **Fact Extractor** reads closed shards with their topic labels as context. Per-shard extraction produces better facts than scanning a mixed-topic message stream. The topic label scopes the extraction: "From the shard about 'Token monitor fix', extract persistent facts."
2. **Channel Compactor** uses closed shards to build Background summaries (Layer 3) — one line per shard, not per arbitrary message window.
3. **Vector Evictor** is unchanged — still sweeps stale hot facts to cold storage weekly.

### 6.3 Cross-Shard References & Hot Memory

When Cortex references an excluded shard ("as we discussed earlier about the executor"), the verbatim conversation is not in context. This is solved by **Hot Memory**, not by pulling back old shards.

The Gardener extracts key decisions and facts from closed shards into Hot Memory. The foreground has the conversational flow of the current shard(s), and the System Floor has the persistent factual context from all past shards via hot facts. The LLM doesn't need the full 80-message conversation about token monitor — it needs "token monitor: TASK column shows evaluator summary, persistent agents show Active, stale cleanup at 2 min" as a hot fact.

`fetch_chat_history` with `shard_id` remains available as an escape hatch — if Cortex truly needs verbatim history from an old shard, it can pull it. But in practice, Hot Memory should cover most cross-shard references.

### 6.4 Background Summaries (Layer 3)

When a channel goes to background, its closed shards provide natural summarization units. Instead of summarizing "the last N messages," the compactor summarizes each recent shard into one line:

```
Background (whatsapp):
- Token monitor fix: implemented TASK column, persistent status, evaluator lifecycle (2h ago)
- Code search tool: built native tool for main agent and Cortex (45min ago)
```

## 7. Configuration

All shard-related config lives under `hippocampus.foreground` in `cortex/config.json`:

```json
{
  "hippocampus": {
    "foreground": {
      "tokenCap": 8000,
      "tolerancePct": 20,
      "maxShardTokens": 8000,
      "timeGapMinutes": 25,
      "semanticCheckInterval": 8,
      "semanticModel": "claude-haiku-4-5"
    }
  }
}
```

| Key | Default | Description |
|-----|---------|-------------|
| `tokenCap` | 8000 | Target foreground token budget for context assembly |
| `tolerancePct` | 20 | Allowed overshoot percentage for including one more shard |
| `maxShardTokens` | 8000 | Token threshold that forces a shard split (Tier 1B) |
| `timeGapMinutes` | 25 | Pause duration that triggers automatic boundary (Tier 1A) |
| `semanticCheckInterval` | 8 | Run semantic topic-shift check every N messages (Tier 2) |
| `semanticModel` | claude-haiku-4-5 | Model used for semantic boundary detection and topic labeling |

## 8. Data Flow (End-to-End)

```
1. INGESTION
   Message arrives → appended to cortex_session → assigned to active shard_id

2. BOUNDARY EVALUATION (async, behind the conversation)
   Tier 1A: Time gap ≥ 25 min since last message? → close shard, open new
   Tier 1B: Active shard tokens ≥ 8,000? → forced split, open new
   Tier 2:  Every 8 messages → Haiku: "did the topic shift?" → if yes, close + label + open new
   (Tier 3: Explicit marker from user/Cortex → close + open new)

3. CONTEXT ASSEMBLY (on each Cortex turn)
   Build foreground bottom-up from closed shards + active shard
   Stop adding shards when token budget (±20% tolerance) is reached
   Include shard separators with topic labels and time distances

4. GARDENER SWEEP (hourly, consumes closed shards)
   Fact Extractor: reads closed shards by topic → extracts facts → Hot Memory
   Channel Compactor: builds Background summaries from shard labels
   (Gardener does NOT create shard boundaries — they already exist)
```

## 9. Migration Path

### 9.1 Phase 1: Schema + Inline Heuristic Detection
- Add `cortex_shards` table and `shard_id` column to `cortex_session`
- Implement Tier 1 heuristic detection (time gap + token threshold) inline
- Messages get `shard_id` on arrival, shards close automatically on heuristic triggers
- No change to context assembly yet — shards are created but not consumed

### 9.2 Phase 2: Shard-Based Foreground Assembly
- Replace unbounded foreground loading with the shard-based assembly algorithm (§4.2)
- Add shard separators to context output
- Configure `tokenCap` and `tolerancePct`
- This is the fix for the 200K token overflow — Cortex becomes bounded

### 9.3 Phase 3: Semantic Detection + Topic Labels
- Add Tier 2 semantic boundary detection (Haiku sliding window check)
- Generate topic labels at detection time
- Add `shard_id` parameter to `fetch_chat_history`

### 9.4 Phase 4: Gardener Integration
- Fact Extractor consumes closed shards with topic labels for scoped extraction
- Background summaries built from shard labels (one line per shard)
- Hot memory bridges cross-shard references

## 10. Open Questions

- **Cross-channel topic continuity:** If the user starts a topic on webchat and continues on WhatsApp, these are separate per-channel shards. Hot Memory should bridge the gap (key facts extracted from the webchat shard), but should there be explicit "related shard" links for fetch_chat_history?
- **Token counting method:** Approximate (chars/4) vs exact (tiktoken-equivalent). Approximate is fast but drifts. The ±20% tolerance band absorbs some drift, but how much? Needs empirical testing.
- **Retroactive sharding:** Should the system process all historical `cortex_session` data on first enable, or only shard forward from activation? Historical sharding provides immediate benefit but could be expensive for large session tables.
- **Semantic check batching:** If multiple messages arrive in rapid succession (burst), should the semantic check wait for the burst to settle, or check after every N messages regardless? Batching might produce better boundary detection.
- **Shard boundary accuracy measurement:** How do we validate detection quality? Manual review of generated shards against expected boundaries? A test dataset of known topic-shift conversations?
